# Reviewer: planner + IR (passes.R, graph.R, node.R, generics.R, chunk_grid.R)

Method: synthetic predict-shaped graph (145×13 + 64×9, 8820×2586 f32, per-year stack → custom band-reduce → maps, two arms, 2-sink multi-export) via load_all; scripts in scratchpad {plan_shape.R, plan_shape2.R, quad_costs.R}. Baseline reproduces handoff plan shape exactly: **2461 source_read stages + 24 compute stages, 24,610 read + 2,736 compute = 27,346 tasks, plan time 7.5 s.**

## Findings (ranked)

- **[architecture] [critical] passes.R:419-421 + lazy_raster.R:73-101** — Per-band SourceNodes force read-task count ∝ bands²; NOTHING in the IR blocks collapsing same-file stack members into one multi-band read — only the executor read path does. The IR already admits it: SourceNode@band is class_integer with NO length validation (node.R:55-73); grid supports outer dims ((band,y,x) is what StackNode's output_grid produces, generics.R:64-72); output_grid(SourceNode) returns node@grid verbatim. Stage exports keyed by node id (passes.R:14-18) → `graph_replace(graph, stack_id, SourceNode(id=stack_id, band=1:145, grid=<stack grid>))` leaves every consumer untouched: a ~30-line planner pass detecting StackNode(along="band") whose parents are all SourceNodes sharing path/open_options/grid. Actual blockers: (a) read task takes scalar node@band (scheduler.R:707-716, executor.R:484-491), GDAL adapter reads one band/window; (b) store-split path writes rank-2 values per compute chunk (store_mb at scheduler.R:758-761 counts only x,y — a (145,y,x) region would be under-budgeted 145×); (c) .plan_read_px counts input NODES not bytes. Impact: ESD 1885 read stages → 13, AEF 576 → 9; read tasks 24,610 → ~220 (112×); also kills pixel-interleave amplification (one task decompresses each strip once for all bands). HIGHEST-LEVERAGE PLANNER CHANGE; aef-multiband-read.md scopes the reader half.

- **[performance] [high] passes.R:781-785 + 787-801** — PROOF no read-window size satisfies both residency and task count at 145 per-band inputs: `tasks >= 8·n_src·n_in·grid_px/B` = 8·2461·145·22.8e6/4.295e9 ≈ **15,100 minimum at any window size** (quantization pushes to 24.6k). One-window-per-source needs B ≥ 26 GB. Per-band: n_src = bands·years, n_in = bands → tasks ∝ bands². Multi-band source: same window size (1916²) but tasks = 22·10 = **220**. Residency and task count decouple only when band count moves out of source count into region shape.

- **[correctness/performance] [high] passes.R:328-334** — Merge-pass reduce-barrier guard checks only the class of the CONSUMED boundary node → one map after reduce_over defeats it and the plan collapses into a mega-stage; demonstrated 18× task inflation. With per-year post-reduce map, consumed root is the MapNode; every 145-input year stage folds into the cross-year join → synthetic collapsed to 2 compute stages with fan-in 1885; compounds through both sizing passes (.stage_bytes_per_px 16,256 B/px → 181² compute chunks, 1470 compute tasks; .plan_read_px cap 285k px → 362² windows, 200/source) = **493,670 tasks vs 27,346** for the identical pipeline. The real predict ("reduce_over → further maps") sits ONE graph-construction detail from this cliff. Fix: guard on the producer stage CONTAINING any Reduce/ScanNode member (or any member whose grid drops an axis), plus cap merged fan-in. 

- **[performance] [high] passes.R:567-581, 612-618, 543-553** — Three O(stages²) scans in finalise cost ~7 s of the 7.5 s plan time at 2,463 stages: (1) D22 out_pad fixpoint Filter per stage per sweep = 6.1M closure calls even with ZERO focal ops (one full sweep always runs); (2) source/warp halo-inheritance repeats full-list Filter per source stage = another 6.1M; (3) exports pass ext_refs <- unlist(all_inputs[-i]) per stage ≈ 15M element copies. Profile: %in% 37% self, unlist 78% total. Scheduler already solved this with consumers_of (scheduler.R:607-614); passes never got it. Fix: build consumers_of + node→stage index once after .merge_stages, reuse in all three. ~7 s → <1 s.

- **[performance] [high] passes.R:508-511 (+413, 501)** — Phase A Find(joinable) scans all protos per node with no compute parent; predicate does %in% over ever-growing closed vector. Per-band pre-stack maps (dequantize shape) TRIPLE plan time to 22.9 s (Find 62.7% total, %in% 47% self). Final plan shape correct — cost is pure scan. Fix: closed as env/logical vector; index open compute stages by sorted-inputs key in an env. ~15 s of host time on the real read→dequantize arm.

- **[performance] [medium] passes.R:742-756 + 699-709** — .plan_chunk_dim abandons block snapping when block is wide (l > 2*side → 1L) → 1-row-strip sources get square windows splitting each strip across 5 x-windows, re-decompressing every strip ~5×. Fix: when native block spans full grid width, prefer full-width row-band windows (8820×420 = 7 windows/source, every strip decompressed once); full-width compute chunk keeps .exec_split_cg integer-multiple alignment. ~4-5× wasted decompress removed. (Convergent with read-path reviewer.)

- **[performance] [medium] passes.R:781-785** — .plan_read_px uses PLAN-WIDE max fan-in: the 145-input ESD stage shrinks the AEF arm's windows too (AEF at own n_in=64 → 4 windows/source vs 10; 2,304 vs 5,760 AEF read tasks). Fix: size per source stage from max n_in over ITS consumers. ~14% fewer read tasks; palliative — finding 1 is the real fix.

- **[architecture] [medium] passes.R:735-756** — One plan-global compute chunk_dim sized by the WORST stage's bytes/px penalises all others (479² for all 24 stages because the 145-input stages need it; 13-input join stages could run at full 1e6-px target with 4× fewer tasks; in the mega-stage failure the global dim drops to 181² for everything). Fix: per-stage chunk dims constrained to integer multiples of a base tile, reusing the split machinery (executor.R:279-306 proves compatibility). FLAG: D14 is a locked decision. ~2-4× fewer compute tasks on the narrow predict tail.

- **[correctness] [verified, no defect] D22 halo machinery (passes.R:166-230, 566-618)** — Looked for superlinear halo inflation; none. need(S) = halo + out_pad additive per fixpoint, matching halo-propagation.md:104-108; runtime pad walk and static mirror implement identical recurrence, cannot diverge; reduce_partial can never receive out_pad > 0 (combine's nonzero pad refused at 582-589) so partial sums cannot double-count. Ring recompute bounded 2r(w+h)/wh per link. Only scale defect is the O(P²) sweep above.

- **[performance] [low-med] passes.R:119-152 + 168-177** — .slim_fn runs codetools::findGlobals (full compiler walk) once per Map/Focal member, uncached: 2,461 dequant maps = 2,461 walks (~1-2 s). Fix: cache by (body identity, environment) — loop-built per-band maps share one body.

- **[performance] [low] graph.R:152-161, 179-180** — graph_import DFS: growing vector + %in% (quadratic) + two S7 property re-assignments per node (each re-running validator); measured 0.96 s @ 8k nodes, ~linear (S7 sets dominate). ~low seconds at 30k. Fix: env seen (as .reachable does) + one-call construction. Only matters when merging foreign graphs (arms usually share one graph; plan_lazy import typically skipped).

- **[ir] [info]** — graph_add is NOT copy-on-append: hashed env, O(1) insert + O(1) bucketed source-dedup (graph.R:200-236); build 1.5 s for 2,531 nodes, dominated by S7/GridSpec construction. graph_toposort O(N²) but test-only cold. Stage/S7 construction at finalise ~11% of plan time; acceptable.

- **[performance] [low] passes.R:719-726, 177** — .stage_bytes_per_px bypasses .node_grid fast accessor, uses real S7 @ over every member (contradicts graph.R:39-43 hot-path rule). ~1% self; cleanup not leverage.

- **[correctness] [low, hypothesis — no concrete evidence] passes.R:382, 643-646** — plan@sink is the stage containing the LAST list element's node; scheduler warp-only elision + compute-on-read fusion exclude only that primary sink (scheduler.R:624, 673); whether a non-primary sink whose stage is a bare source_read consumed only by warps could be elided (losing its sink export) is UNTESTED. Cross-ref: executor reviewer found the same warp_only sink-guard divergence at scheduler.R:620-625. Multi-export spec otherwise faithfully implemented (every sink id forced into exports at 550-551).

## Headline answers
1. Plan shape: 2,461 source_read + 24 compute stages ≈ 27k tasks, matching run 5. Per-band sources forced by lazy_source/lazy_stack CONSTRUCTION, not passes or IR. Missed fusion is not the problem; OVER-fusion is the latent one.
2. No window size satisfies both constraints; only multi-band read breaks the invariant.
3. Four quadratic scans remain in passes (~7-20 s at production scale), all fixable with the scheduler's consumer-index pattern.
4. D22 correct as specified; only scale defect is the O(P²) fixpoint sweep running even for halo-free plans.
5. IR env-based and O(1) where it matters.
