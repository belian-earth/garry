# Reviewer: gdal-direct path (composite_direct.R)

## Verdict on converge-vs-extend: keep two executors, but shift the centre of gravity to the direct path

Do not try to port the direct path's core win into `execute_plan_mirai`. The win is structural, not a mechanism you can graft: one whole-grid f32 buffer per source, warped in place by GDAL (`gdal_adapter.R:355-376`), consumed by one blocking fetch-ordered loop with ~1 task per source and near-zero host bookkeeping (`composite_direct.R:496-587`). The scheduler's cost is equally structural: ~20k (stage, window) tasks, each with a mori region create/map/refcount/unlink cycle and a polling host loop (`scheduler.R:633-990, 1168-1271`); the two portable mechanisms have in fact already been ported (raw `mirai()` by-name dispatch was always the scheduler's pattern, `scheduler.R:779-787, 905-918`; window-major fetch-ordered launch is in the uncommitted work, `scheduler.R:1082-1105`). Four consecutive scaling defects in the scheduler versus a direct path whose blockers are mostly eligibility checks is the decisive evidence. Recommendation: widen the direct whitelist to cover the predict family (multi-export, custom reduce fn, band axis, local non-GTI sources, unary pre-map chains: each individually cheap, itemised below), and fix the direct path's one genuine structural gap first, its whole-run tmpfs/buffer lifecycle, which does not fit the cohort as written. Keep `execute_plan_mirai` as the fallback for what direct cannot express (ScanNode graphs, warp barriers, global reduce_combine, gradient) rather than as the path production traffic rides. Evidence gap: run 5's slowness is undiagnosed; this review argues from code structure and arithmetic, not a cohort-scale measurement of the extended direct path.

## Findings, ranked by leverage

**Why the predict graph does not qualify (Q1).** The graph is disqualified five times over, in this order:

- **[eligibility] [critical] collect.R:51-59** — Multi-export plans are routed to `execute_plan_mirai` before the direct path is even consulted; evidence: the `length(p@sinks) > 1L` branch returns at line 51-71, while `.cd_spec`/`.gd_decompose` are only reached in the single-sink branch at collect.R:83-94, and the failing predict step is one multi-export `collect()`. Proposed change: consult `.gd_decompose` per sink over the shared graph, fetch each source once (bins are already keyed by node id, composite_direct.R:338), and run one upper kernel per sink; `.compose_stage_fn`'s `exports` argument is already plural (passes.R:166, 207-209) so the upper stage generalises directly. Impact: this is the gate; nothing else matters until it opens.
- **[eligibility] [critical] composite_direct.R:611-615 (and duplicated at 198-202)** — Every source must have a `GTI:` path plus a `.meta.rds` sidecar, which the cohort's local per-band GTiffs/VRTs (hutan `embed_read` outputs) do not have; evidence: `grepl("^GTI:", n@path)` and `file.exists(paste0(..., ".meta.rds"))` both hard-fail `.gd_spec`, yet `gdal_warp_to_buffer` accepts any GDAL-readable source and the meta sidecar exists only to enumerate remote items per slice. Proposed change: in `.gd_build_jobs` (composite_direct.R:322-348) add a non-GTI branch where `locs = n@path`, `dt = 0`; nothing downstream cares. Impact: unlocks local-file workloads for a day's work; also removes the drift risk of three copies of the fetchability predicate (the third is `prepare_fetch`, scheduler.R:547-556, which *falls back* instead of refusing, a semantic divergence already).
- **[eligibility] [high] composite_direct.R:151-153** — `.cd_reduce_spec` rejects custom reducers: `red@op %in% c("median","mean",...)` fails because `reduce_over(x, fn, ...)` stores `op = "custom"` (lazy_raster.R:396-403). Extension is safe and small: the compute-daemon task bodies already load anvl (`.require_anvl()` at composite_direct.R:82, 107, 128) and run on the `garry_compute` profile (lines 542, 577), the same daemons that run `mlp_project` under the scheduler, so ship the fn in `Kb` and call `fn(m, 1L)` in place of `.apply_reduce(k$op, m, 1L, ...)` (lines 138, 141). One caution: do **not** `.slim_fn` a custom reducer; `.compose_stage_fn` deliberately leaves them unslimmed because factory bodies (band_project, mlp_project) resolve garry internals through their namespace-parented environment (passes.R:172-175), whereas `Kb` currently slims `F` (composite_direct.R:549). So the direct path is not reduce-only: it has a real compute-daemon stage, and the MLP belongs on it.
- **[eligibility] [high] composite_direct.R:152, 715** — Only `"t"` reduces qualify (`"t" %in% red@over` in `.cd_reduce_spec`; `is_red_t` in `.gd_decompose`), but `mlp_project` reduces `over = "band"` (band_mlp.R docs, lazy_raster.R:480). The kernels are already axis-agnostic: the cube is built `(n_slices, ny, nx)` and reduced positionally on margin `1L` regardless of the axis name (composite_direct.R:93, 138). Relaxing the check to "any single leading axis" is a two-line change plus a `dims` mapping for custom fns. Impact: with the previous item, the whole `stack -> reduce_over(mlp)` shape lifts.
- **[eligibility] [medium] composite_direct.R:159-176** — A unary map chain between a band source and the stack (e.g. dequant scale/offset, which `embed_read` sources carry) fails both arms of `.cd_reduce_spec`: arm 1 requires the slice to be a 2-parent masked MapNode (line 160), arm 2 a bare SourceNode (line 173). Proposed change: allow a linear unary Map/Focal chain per slice, replayed cube-vectorised exactly as `.cd_walk_mask`/`.gd_replay_mask` already do for the mask (lines 27-62). Impact: needed for any harmonised/scaled source; without it the extended whitelist still misses real pipelines.

**The structural gap that must be fixed before extending.**

- **[architecture] [critical] composite_direct.R:521-522, 486, 768** — The pipeline dispatches every band's fetch up front and never frees a source `.bin` until end of run (the only cleanup is the `on.exit(unlink(tmp, ...))` on the whole tmp dir), so tmpfs residency is the entire plan's source set; evidence: at cohort scale that is roughly (1885 + 576) full-grid f32 slices x 91 MB ≈ 220 GB on /dev/shm, and each extended "band" task's working set is 3.5 x 145 x 91 MB ≈ 46 GB by the path's own estimate (`.gd_compute_cap`, lines 296-303); the code admits it: "Whole-grid (fits in memory); spatial chunking for scale is the next stage" (lines 596-598). Proposed change, two stages: (a) window the fetch dispatch (launch band b+W's fetches when band b harvests) and unlink a band's bins in `harvest()` (lines 560-567), bounding residency to W x one-band cube; (b) spatial chunking of the whole direct pipeline for grids/cubes that exceed RAM. Impact: (a) alone makes the cohort feasible on a large box; without it, extending the whitelist trades the scheduler's host stall for a guaranteed OOM.
- **[performance] [high] gdal_adapter.R:355-376; composite_direct.R:322-348** — Neither executor fixes the pixel-interleave amplification: the direct path warps one band per whole-grid task, so a 145-band 1-row-strip DEFLATE year file is fully decompressed up to 145 times per year (256 MB block cache, gdal_adapter.R:409, cannot hold a multi-GB year file); evidence: `BANDS=1` in the MEM DSN at line 366, one job per source node at lines 344-346. Proposed change: a multi-band warp job (one `.cd_fetch_warp` per (file, slice) producing N bins, `BANDS=N` MEM DSN with interleave handling), which the direct path's job model accommodates far more easily than the scheduler's task table; same conclusion as handoff 3.2 and design/aef-multiband-read.md. Impact: roughly two orders of magnitude fewer read tasks and up to ~145x less decompression; likely the single largest lever on cohort wall time in either executor.

**Correctness of the direct path itself.**

- **[correctness] [high] composite_direct.R:261-278 vs 380-395 and 420-427** — A failed warp fetch silently becomes an all-NaN slice: `.cd_fetch_warp` catches every error into `r$err` and always writes a complete buffer ("real or all-NaN", line 277), but `.gd_warp_collect` and `.gd_check_fetch` inspect only `inherits(x, "miraiError")`, so the `err` field is write-only and even the warning never fires for caught errors. Silently overrides `garry.read_fail = "error"` (options.R:51), which the scheduler honours (`scheduler.R:171-173`). Fix: collect `r$err`, abort under `read_fail = "error"` / warn under `"nodata"`. At 2.4k fetches transient vsicurl/IO failures WILL occur and today produce plausible-looking composites with holes and no diagnostic. Predates e24e8ef, but e24e8ef added more unchecked collection sites (lines 569, 574).
- **[correctness] [medium] composite_direct.R:165-171** — Per-slice homogeneity assumed, not checked: `F` lifted from slice 1 (`first@fn`), mask chain from slice 1 (`m0$chain`); other slices only checked for walkability + fmask source id. Different per-slice fns silently computed with slice-1 semantics. Fix: compare serialized slimmed fn/chain per slice against slice 1 (`.gd_hash` machinery exists at 696-699); return NULL on mismatch.
- **[robustness] [medium] composite_direct.R:161, 167** — Eligibility probing can crash instead of falling through: `gg(id)@parents[[1L]]` on a bare SourceNode slice or `@parents[[2L]]` on a unary MapNode subscript-errors out of `collect()` when a stack mixes masked and unmasked slices. Guard with length checks or tryCatch-to-NULL.
- **[correctness] [low] composite_direct.R:574, 563-566, 486** — Mid-pipeline failure ragged: `mask_p[]` never checked (failed mask compute surfaces as cryptic readBin error); `harvest()` abort abandons in-flight tasks while `on.exit(unlink(tmp))` deletes dir daemons may be writing into.
- **[correctness] [info]** — D13/mosaic pinning/band order clean as claimed: MEM mechanics quarantined in adapter (gdal_adapter.R:332-376), overlap ordered by datetime matching GTI SORT_FIELD, band output order follows top@parents, NaN pad matches D8, whole-grid = no seams so no D22 needed. No defect found.

**Duplication and drift.**

- **[drift] [medium] composite_direct.R:44-62 vs passes.R:57-77** — `.gd_replay_mask` hand-reimplements `.eval_node`'s focal stencil, tethered only by a comment; ignores `FocalNode@boundary`. Extract shared stencil helper.
- **[simplification] [medium] collect.R:83-94; composite_direct.R:601-673** — `.execute_gd_general`/`.gd_spec` unreachable from `collect()` (only exercised by test-gd-general.R); duplicates materialise-and-write tail. Wire as fallback or delete.
- **[simplification] [low] composite_direct.R:406-418, 660-673, 794-803** — Three near-identical materialise/write tails; fold into `.gd_write_result`.
- **[simplification] [low] composite_direct.R:229-236 + options.R:86-92, 118** — `.cd_spec` heavy-composite routing gate dead under defaults (`gd_parallel` TRUE means `gd_compute_budget` never routes). Delete or re-justify.
- **[performance] [low] composite_direct.R:775-780** — `.execute_gd_reduce` runs groups sequentially; group 2's fetches start after group 1's compute drains. At cohort scale ESD/AEF arms serialise two multi-minute fetch phases. Overlap next group's fetches with current compute.

**Scheduler-side portability note.** The one direct mechanism worth porting INTO the scheduler: warp-on-read as a read-task kind — replace per-item fetch + assemble pair (`scheduler.R:546-604, 703-842`) with a `.daemon_run_source_shm` variant calling `gdal_warp_to_buffer`, halving read-side task count and deleting the tmpfs GTiff round-trip. The preallocated full-grid buffer model is NOT portable (incompatible with chunked back-pressure and refcounted store, which exist for plans that don't fit in memory).
