# Reviewer: API layer and periphery (lazy_raster, dataset, stac, preview, draw, gradient, scan_kalman, band_mlp, options, zzz)

Measurements from scratchpad graph_build.R and mlp_bench.R (devtools::load_all, this machine). Ranked by leverage on the bc-cohort failure first.

- **[architecture] [HIGH] collect.R:51-71** — A multi-export collect() is structurally barred from BOTH fast paths: the multi-sink branch dispatches straight to execute_plan_mirai (51-59); .cd_spec and .gd_decompose are only consulted in the single-sink branch (83-92). The answer to handoff §3.3's "why does the predict graph not qualify": eligibility is never even checked when length(p@sinks) > 1. Fix: run .gd_decompose per sink (each sink's subgraph is reduce-structured), sharing the read plan across sinks; or at minimum cli-log why a plan fell to the general path. This is the single decision point keeping the workload off the ODC-parity architecture.

- **[performance] [HIGH] gdal_adapter.R:409** — GDAL_CACHEMAX hardcoded 256 MB per read daemon, no garry option; breaks the block-cache mitigation at 145 bands. One decompressed all-band row of the 8820-px grid ≈ 5 MB (145 × 8820 × 4 B); 256 MB caches ~50 rows vs ~420-row budgeted windows → by a daemon's second band of the same window every strip's blocks are evicted; whole window re-inflates. With ~145 band-tasks per window over independent daemon caches, amplification approaches ×145 — daemons busy, wall terrible, matching run 5. Sizing argument, not measurement; verify by rerunning with GDAL_CACHEMAX ≈ window_rows × 5 MB (~2 GB) on read daemons. Fix: make it a garry_opt sized from planned window + band fan-in in garry_daemons() (scheduler.R:423-434); cross-ref the real fix (multi-band read tasks).

- **[performance] [MEDIUM — mostly good news] band_mlp.R (mlp_project)** — correct and cheap when cached; NOT the per-task bottleneck. Measured [145,256,256] chunk: traced+jit warm 12.8 ms/chunk, first-compile 21 ms, new-shape 51 ms, pure-R oracle 741 ms; traced vs oracle max diff 8.1e-7; NaN complete-cases semantics verified identical. ~4.5 s single-core compute per year-stage at 22.8 Mpx. Weights land as f32 constants (anvl promotion.R:66-77), no f64 promotion, no per-call upload. Cheap win: fold center/scale into layer-1 weights (band_mlp.R:88-95) — marginal under XLA fusion.

- **[performance] [MEDIUM] scan_kalman.R** — working set ~6× the planner's per-pixel memory model for scan stages: kalman_llt casts f64 (93-94), forward scan emits seven (T,y,x) f64 cubes (140-144) live until backward pass (165-174): ~730 B/px at T=13 vs the model's ~128 B/px (passes.R:719-726); robust passes re-run the smoother per iteration. A 1 Mpx chunk holds ~730 MB live vs ram_budget_mb 512. Fix: per-node memory multiplier hook (ScanNode reports own bytes/px) or documented scan chunk cap. Flagged from code, no runtime OOM measured. Otherwise clean: f64 cancellation notes respected end-to-end, cancellation-free RTS forms match header derivation, robust Q-timing shift correct, fixed-count-equals-early-exit argument sound, no per-pixel R loops.

- **[performance] [GOOD NEWS] Graph construction via public API is linear, not O(n²).** 145 bands × {9,17,34} years = 0.60/0.99/2.03 s for 1.3k/2.5k/5k source nodes shared-graph (~0.4-0.7 ms/node). graph_add O(1) env insert; D6 dedup bucketed; lazy_source with grid= skips GDAL; LRU handle cache makes 145 same-file probes cheap. Dataset step recording O(1)/verb. API-layer growth is NOT the bottleneck.

- **[performance] [MEDIUM] draw.R:81-98** — draw() is EXPONENTIAL in shared-parent depth: .ir_tree recurses once per path, no memoization. Diamond DAG measured: depth 10 = 0.30 s, 13 = 2.23 s, 16 = 18.25 s, doubling per level; depth 22 DNF in 120 s. Hangs on exactly the graphs being debugged. Fix: memoize rendered subtree by node id; bottom-up signatures, O(nodes).

- **[correctness/architecture] [MEDIUM] dataset.R:215-219** — `[[` on LazyDataset mints a new StackNode per access, no CSE: ds[["nir"]] - ds[["red"]] with a band referenced twice builds two identical stack subtrees; planner computes both (reads dedup, stacked compute does not). Every access also mutates the shared graph. Fix: cache stacked node id per band on the dataset, or extend dedup index to StackNode(parents, along). Silent 2× on natural band math.

- **[options] [MEDIUM] options.R:41 + passes.R:783** — read_budget_mb 4096 with per-band sources forces ~22k tasks; the model double-charges same-file bands (divide-by-n_in assumes independent reads, but same-file interleaved bands share decompression and could share one region). Buys OOM safety with exactly the task count runs 3-5 choked on. No knob decouples read-task granularity from residency. Real fix: multi-band read tasks; meanwhile document n_in scaling and expose the hardcoded double-buffer factor 2.

- **[footgun] [MEDIUM] preview.R:277-282, 301-303** — preview() of the SI pipeline silently launches the FULL workload: .preview_coarsen returns NULL unless every source carries RESX= open options; bc per-band sources do not → full-res collect() of 22.8 Mpx × 145 bands to draw a thumbnail. Fix: cli-warn with projected pixel count + opt-in threshold, or synthesize coarse read via WarpNode. Related [LOW]: .pv_read_path reads whole files full-res (no overviews; 3-band 22.8 Mpx preview ≈ 550 MB vs ~1 MB overview read) — pass decimated out sizes. [LOW]: .coarsen_node drops source resampling (bilinear previews as near).

- **[performance] [LOW] graph.R:89-117** — graph_toposort quadratic (named-vector match + c() appends): 0.14/0.48/2.16 s at 1.3k/2.5k/5k nodes. Latent: only tests call it (planner uses O(n) .reachable). Fix or un-export before a future caller inherits it.

- **[options] [LOW] Missing/awkward knobs at production scale.** (a) progress = FALSE default: every diagnostic round needed liveness; default TRUE for distributed or auto-enable above N tasks. (b) Hardcoded scheduler constants debugging kept touching: 2 ms poll sleep (scheduler.R:1270), gc-throttle cadence (957-976). (c) Daemon counts are function args only, not options — Rscript pipelines cannot tune pool sizes via options(). (d) read_handles forced 1 (scheduler.R:409,428): reopen churn at 20k tasks over a handful of local files (weak evidence, cheap to raise). No dead options: all 18 defaults consumed.

- **[quality] [LOW] Non-cli errors in public entry points** (house-rule violations): stopifnot() in lazy_map (lazy_raster.R:128), lazy_stack (161), focal_kernel (534), kalman_llt (scan_kalman.R:87), stac_sources (stac.R:155), stac_gti_index (470), stac_filter_coverage (209), lazy_value_and_grad (gradient.R:160-161). Mechanical.

- **[performance] [LOW] stac.R + gti_index_create** — per-band setup linear but host-serial: 145 index files written feature-by-feature + one gdal_grid_spec probe per asset (~0.1 s remote, serial) ≈ a minute at HLS-like counts. Not on bc path. Band harmonization vectorized, no defect.

- **[quality] [INFO] zzz.R and gradient.R sound.** .onLoad registers S7 methods, cli-warns GDAL < 3.9; GDAL config deliberately off host session (scheduler.R:415-420); mirai owns pool lifecycle. gradient.R tape rejections precise and typed; only inefficiency is cfn re-evaluating the chunk pipeline separately from vg per chunk (176-205) — fusable, not on failing path.
